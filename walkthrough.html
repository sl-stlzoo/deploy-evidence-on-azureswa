Walkthrough

Here are the steps we'll walk through to deploy.

1. Prerequisites

For this workshop, the recommended environment is a GitHub Codespace. This eliminates the need to install local tooling like Node.js, Azure CLI, or the SWA CLI.

Before launching the environment, you must configure GitHub Codespace secrets in your repository settings:

  • AZURE_SUBSCRIPTION_ID: Your Azure Subscription UUID.
  • AZURE_TENANT_ID: Your Microsoft Entra ID Tenant UUID.
  • AZURE_LOCATION: Target Azure region for deployment (e.g., eastus2).
  • AZURE_RESOURCE_GROUP: Name for the resource group to create.
  • SWA_NAME: A globally unique name for your Static Web App.
  • AAD_APP_NAME: A display name for your Entra App Registration.
  • REPO_URL: The HTTPS URL of this repository.

You also need an active Azure Pay-As-You-Go subscription with Contributor access, and Application Administrator privileges in your Entra ID tenant to create app registrations.

2. Initializing Evidence.dev

Instead of committing the Evidence.dev template directly to our repository, we pull it down dynamically. When the Codespace starts, a lifecycle script (post-create.sh) automates the scaffolding process:

  1. Calls degit to clone the base Evidence.dev template dynamically, ignoring Git history.
  2. Runs npm install to fetch all required Evidence dependencies.
  3. Installs Azure tooling (SWA CLI and Azure CLI extension) inside the Codespace.
  4. Maps your GitHub Codespace secrets into a deploy/.env file to prepare the automation pipeline.

This approach ensures you always launch with the latest Evidence.dev framework without polluting your repository with upstream boilerplate.

3. Azure Static Web App Provisioning

To deploy the app, you will execute a single orchestration script from the VS Code terminal: .\deploy\deploy.ps1.

During the provisioning phase, the PowerShell automation will:

  • Verify that your Azure CLI is authenticated and connected to the right subscription.
  • Create the designated Azure Resource Group if it doesn't already exist.
  • Provision an Azure Static Web App at the Standard SKU via Bicep limits or Azure CLI (note: the Standard tier is required for custom App Registrations and Role-Based Access Control integrations).

The provisioning scripts are written to be idempotent. You can re-run them safely without breaking existing infrastructure.

4. GitHub Actions Workflow Structure

After provisioning the resources, the pipeline sets up continuous deployment using GitHub Actions.

The deploy script captures the required deployment tokens from your new Azure Static Web App and generates a `.github/workflows/` YAML file tailored for Evidence.dev.

Once you commit and push the updated repository, GitHub Actions will take over. The workflow uses the standard SWA action to run the Evidence build step (npm run build) and subsequently publishes the resulting build/ directory and any managed Azure Functions backend APIs directly to the CDN edge nodes.

5. Securing with Microsoft Entra ID Authentication

One of the primary goals of this architecture is restricting access to internal teams using Entra ID (Azure AD). The deployment orchestrator configures this in multiple layers:

  • App Registration: Generates a single-tenant App Registration locking the audience constraints strictly to your organization's directory.
  • Routing Protections: Updates the staticwebapp.config.json with a system-wide redirect to /.auth/login/aad—blocking all unauthenticated access.
  • Custom Roles via Backend API: Integrates a managed Azure Function (api/GetRoles). Upon user login, SWA sends claims to this function, which verifies the user matches your active tenant. If successful, it assigns the custom evidence_user role required to access the site content.

For sensitive insights, this provides full defense-in-depth, seamlessly integrating Single Sign-On (SSO) onto an otherwise static, headless frontend.
